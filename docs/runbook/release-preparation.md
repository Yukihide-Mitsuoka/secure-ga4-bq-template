# Prepare and verify a GitHub Release

**Trigger:** the repository has releasable commits but no current Release PR, or a human
has merged a Release PR and needs post-release verification.

**Impact if unhandled:** releasable changes remain unversioned, or a published release
can lack required test, vulnerability, SBOM, build, or provenance evidence.

**Safe to execute by agent:** partially. An agent may enable and dispatch the workflow,
audit the generated PR, and verify a completed release. The agent must stop before
merging the Release PR because REL-010 reserves release approval for a human.

## Preconditions

1. Confirm `main` has no active implementation PR and the latest CI, Security, and
   CodeQL runs are green.
2. Confirm the task does not require a cloud environment. Release preparation uses only
   GitHub repository metadata and the checked-in source.
3. Authenticate GitHub CLI as a repository administrator:

   ```bash
   gh auth status
   ```

   - Expect: the target account is authenticated for `github.com`.
   - If not: stop and ask the repository owner to authenticate.

## Prepare the Release PR

1. Enable the checked-in workflow:

   ```bash
   gh workflow enable Release --repo Yukihide-Mitsuoka/secure-ga4-bq-template
   gh workflow view Release --repo Yukihide-Mitsuoka/secure-ga4-bq-template
   ```

   - Expect: `Release` reports an active state.
   - If not: verify Actions administration permission; do not replace the workflow token
     or add a personal access token without a separate security review.

2. Dispatch the workflow against the default branch:

   Before dispatch, record the latest published release so a preflight cannot silently
   change release state:

   ```bash
   gh release list \
     --repo Yukihide-Mitsuoka/secure-ga4-bq-template \
     --limit 1 \
     --json tagName,publishedAt
   ```

   ```bash
   gh workflow run Release \
     --repo Yukihide-Mitsuoka/secure-ga4-bq-template \
     --ref main
   ```

   - Expect: one queued `workflow_dispatch` run.
   - If not: confirm the workflow is active and `main` contains the
     `workflow_dispatch` trigger.

3. Watch the dispatched run:

   ```bash
   RUN_ID="$(gh run list \
     --repo Yukihide-Mitsuoka/secure-ga4-bq-template \
     --workflow Release \
     --event workflow_dispatch \
     --limit 1 \
     --json databaseId \
     --jq '.[0].databaseId')"
   gh run watch "$RUN_ID" \
     --repo Yukihide-Mitsuoka/secure-ga4-bq-template \
     --exit-status
   ```

   - Expect: `release-please` succeeds and `release-gates` runs against the dispatched
     `main` SHA. The gate must complete the full test suite, vulnerability scan, SPDX
     and CycloneDX SBOM generation, build, and provenance attestation.
   - The two SBOMs are workflow artifacts named `sbom-spdx` and `sbom-cyclonedx` during
     preflight. They become GitHub Release assets only after a Release PR is merged.
   - A manual dispatch may create or update the Release PR, but it must not merge that
     PR, create a tag, or publish a GitHub Release.
   - If not: inspect `gh run view "$RUN_ID" --log-failed`; fix the workflow through a
     normal PR. Never hand-edit generated version or changelog content.

4. Confirm preflight did not publish a release:

   ```bash
   gh release list \
     --repo Yukihide-Mitsuoka/secure-ga4-bq-template \
     --limit 1 \
     --json tagName,publishedAt
   ```

   - Expect before the initial release: no releases are listed.
   - On later releases, expect the existing latest release to remain unchanged. Compare
     its tag with the value recorded before dispatch.

5. Locate the generated Release PR:

   ```bash
   gh pr list \
     --repo Yukihide-Mitsuoka/secure-ga4-bq-template \
     --search 'is:open label:"autorelease: pending"' \
     --json number,title,url,headRefName
   ```

   - Expect: exactly one open Release PR.
   - If not: inspect the successful run summary before dispatching again.

6. Audit the Release PR according to `.ai/release.md`:

   - Compare every generated changelog entry with the Conventional Commit history.
   - Confirm the SemVer bump accounts for all `feat`, `fix`, and breaking commits.
   - Confirm `version.txt` changed from the unreleased `0.0.0` sentinel to the proposed
     release and that `CHANGELOG.md` was generated by Release Please.
   - Confirm only release-please-owned version and changelog files changed.
   - Record current main-branch test, CodeQL, vulnerability, secret, and license results.
   - Record that this repository publishes no container image, so the container-image
     gate is not applicable.
   - Summarize blast radius and rollback: before merge, close the Release PR; after
     release, use a reviewed `revert:` commit and prepare a patch release.

7. Stop and request human approval. The agent must not merge the Release PR.

## Verify after human approval

1. Watch the `push` run created by the human merge:

   ```bash
   gh run list \
     --repo Yukihide-Mitsuoka/secure-ga4-bq-template \
     --workflow Release \
     --event push \
     --limit 1 \
     --json databaseId,status,conclusion,url
   ```

   - Expect: `release-please` and every `release-gates` step succeed.
   - If not: do not delete or move the tag. Fix forward on `main` and prepare a patch
     release.

2. Verify the GitHub Release and required SBOM assets:

   ```bash
   TAG="$(gh release list \
     --repo Yukihide-Mitsuoka/secure-ga4-bq-template \
     --limit 1 \
     --json tagName \
     --jq '.[0].tagName')"
   gh release view "$TAG" \
     --repo Yukihide-Mitsuoka/secure-ga4-bq-template \
     --json tagName,isDraft,isPrerelease,assets,url
   ```

   - Expect: a non-draft, non-prerelease tag with `sbom.spdx.json` and `sbom.cdx.json`
     assets; the run also contains successful test, vulnerability, build, and provenance
     steps.
   - If not: retain the immutable release record and report the missing evidence. Do
     not manually backfill assets, delete the release, or move its tag; fix forward
     through a reviewed patch release and verify that release independently.

## Escalation

Escalate to the repository owner when the generated version is unexpected, a breaking
change lacks migration notes, any REL-020 gate fails, the workflow needs a broader token,
or a published tag would need deletion or movement.
