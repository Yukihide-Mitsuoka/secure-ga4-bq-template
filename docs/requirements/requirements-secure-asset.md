## 1. エグゼクティブサマリ

- **何を**: セキュリティ統制3項目（①列レベルセキュリティ〔ポリシータグ〕②IAM最小権限 ③コスト最適化監査ログ）を組み込んだ、GA4→BQ **マート層** のセキュア標準アセット。**構築モード**（Terraform＋CI/CDでセキュアに建てる）と**点検モード**（既存環境を全点検し是正提案する）の2モード。
- **前提となる設計判断**: 統制は自作の**マート層**に効かせる。生の `analytics_*` エクスポートはGoogleが日次生成するシャード／スキーマドリフトと宣言的管理が構造的に噛み合わないため、**データセット単位IAMで封鎖**する。
- **誰に**: 中堅のデータ活用企業（GA4を大量利用するが、セキュリティ専任チームは薄い層）。
- **どう売る**: 受託×再利用アセット。点検定額（30〜50万円）で入り、構築（人日）へ展開。2案件目以降はテンプレ再利用で工数削減。（事業設計の正本は [型化文書](requirements-service-packaging.md) §2）
- **差別化**: GA4→BQ特化の深さ（機密列カタログ・ネスト展開設計・監査ログのコスト設計）。汎用ガバナンスベンダーにも汎用SIにも無い部分。

## 3. 背景・目的とスコープ

### 3.1 背景・目的
GA4→BQの従来構築（エクスポート→クエリ→マート→可視化）はセキュリティ統制を含まない。本アセットの価値は「**同等以下の工数で、従来ゼロだった統制を組み込む／既存環境を点検して是正する**」こと。速さではなく "時間を増やさず安全を足す"。

### 3.2 対象統制（3項目）
- ① BQ列レベルセキュリティ（ポリシータグ／タクソノミー）
- ② IAM最小権限（データセット／テーブル単位）
- ③ コスト最適化された監査ログ設計

### 3.3 適用層
- 対象: 自作の**マート層**（および中間層）。自分で作る＝宣言的管理下＝タグ／ポリシーが安定して効く。
- 封鎖: 生 `analytics_*` はデータセットIAMで塞ぐ。列レベル制御の対象外。

### 3.4 非スコープ（明示的に外す）
- 行レベルセキュリティ（今回は対象外）
- Cloud DLP を前提とした保護（高コストのため前提にしない。低コストなポリシータグを基本）
- 可視化ツール（Looker等）側のアクセス制御
- **送信側（収集時）のPII混入防止**（タグマネ/クライアント計測設定の是正）＝別テーマ・対象外。
  - 区別: **BQ格納後のマート列の値スキャンによるPII混入検知**は対象内で、A+観点として扱う（FR-4）。「送信を止める」話は外し、「入ってしまった値を見つける」話は上振れとして持つ。

## 4. 動作モード

### 4.1 構築モード（build）
Terraform＋CI/CDで、統制3項目を組み込んだマート基盤をクリーン構築する。
- マートを **dbt / Dataform で建てる場合の実装レール**（`policy_tags`/partition/clusterを変換ツールのconfigで宣言、CIでdry-run予算ゲート）を選択可能な経路として持つ → 詳細は [requirements-dbt-dataform-rail.md](requirements-dbt-dataform-rail.md)。
- 変換ツールを使わない素のSQL＋Terraform経路も許容する。

### 4.2 点検モード（inspect）
導入済み環境を全点検し、リスク一覧をレポート化（カバレッジ100%）する。
- **カバレッジ100%の母集合定義**: 対象プロジェクト内の「マート/中間データセット、そのテーブル・列、関連IAMバインディング、監査ログ設定」を分母とする。生 `analytics_*` は封鎖確認（IAMで塞がれているか）のみを対象とし、列単位点検の分母には含めない。
- IAM過剰権限は最小権限の是正設計（推奨ロール／Terraform）を提示。IAM以外の是正可能項目は是正または是正手順を提示。
- 生成AIで自然言語レポートと是正用Terraform/Policyのドラフトを自動生成する（＝AIは "決定論的な点検結果という枠内で" 書く）。

## 5. 機能要件

### FR-1 タクソノミー／ポリシータグ管理（構築）
- タクソノミー定義（機密度レベル: **`high`/`medium`/`low` の3段**）をTerraformで宣言。
- マート列への `policy_tags` 付与を宣言的に管理（変換ツール利用時はそのconfigに委譲）。
- データポリシー（列マスキング）を任意で付与できる。

#### FR-1.1 GA4固有 機密列カタログ（差別化の中核）
GA4エクスポート由来のマート列を機密度に写像する再利用カタログ。GA4非特化の競合が持てない部分。

| 機密度 | 該当（GA4由来の代表列） | 扱い |
| --- | --- | --- |
| **high** | `user_id`（自社設定の会員/CRM ID化が多い）、`event_params` の値でメール/電話/生ID等のPII、`page_location`/`page_referrer` のクエリ文字列に混入するPII、`user_properties` の個人特定値 | マスキング/限定アクセス。PII混入は検知的統制と併用（FR-4） |
| **medium** | `user_pseudo_id`（擬似ID・分析結合に必要）、`geo.*`（city粒度）、`device.*` 詳細、`session`/`traffic_source` 系 | 準識別子。閲覧はロール限定 |
| **low** | `event_name`、`event_date`/`event_timestamp`、集計向けディメンション | 制限なし |

> 注: `page_location` 等のクエリ文字列PIIは列レベル制御だけでは防げない（値に混入するため）。カタログで high 指定しつつ、FR-4の点検で混入検知を回す＝GA4特化の勘所。

#### FR-1.2 カタログは上書き可能なパラメータ入力（拡張性要件）
機密度カタログは**固定値ではなく、案件ごとに上書き可能なパラメータ入力**として設計する。
- 既定マッピング（上表）を「推奨デフォルト」として同梱し、案件側の設定で列単位に上書きできる（例: `user_pseudo_id` の既定= `medium`、規制要件の高い案件では `high` に昇格）。
- 昇格/降格はコード改修なしに設定変更で完結すること（テンプレ本体は不変、案件差分は入力で吸収）。
- ローンチ後の運用・顧客反応を受けて既定値やレベル定義を更新できる余地を残す。
- 実装示唆: マッピングを外部設定（YAML/変数）化し、構築（タグ付与）・点検（未設定/不整合検知）の双方が同じ設定を参照する。

#### FR-1.3 ネスト列（event_params/user_properties）の展開設計
`event_params`/`user_properties` は key-value のネスト（`value.string_value` 等が単一列に同居）のため、ネストのままでは特定キーだけを列レベル保護できない。
- **方針: staging→mart の変換で必要キーを型付き列に unnest し、その自作列に `policy_tags` を付与する**（＝マートを守る思想の具体化）。展開列は宣言的管理下なのでタグ・マスキングが安定して効く。
- 展開するキーは「案件パラメータ」で宣言（FR-1.2）。機密キー（メール/電話/生ID等）は展開列を high に。
- 併用可: **allowlist**（承認キーのみマートに取り込み、残りは落とす/低信頼の生ネスト列に留置）でプライバシーを強化。
- 生ネストのまま残す値のPII混入は FR-4 A+ の値スキャン検知で補完（列制御と検知の二段）。

### FR-2 IAM最小権限（構築）
- basic roles（`roles/owner`,`roles/editor`,`roles/viewer`）をプロジェクト付与しない設計。
- データセット／テーブル単位の事前定義ロール（`roles/bigquery.dataViewer` 等）＋条件付きで付与。
- サービスアカウント／グループ単位でロールを宣言。

### FR-3 監査ログ4層設計（構築）
1. 通常監査は `INFORMATION_SCHEMA.JOBS` で代替（Data Accessログの取り込み課金を避ける）
2. Data Accessログは**高感度データのみ**有効化
3. Cloud Logging 除外フィルタでノイズを落とす
4. Log Router で BQ／GCS へルーティング＋保持短縮

### FR-4 点検観点セット（点検）※B基準の測定対象

**B観点＝決定論的・高信頼（11観点を定義、B基準は10観点以上の検出）。** 値の中身を見ずメタデータ/設定で判定できるもの。

| # | カテゴリ | 観点 |
| --- | --- | --- |
| 1 | IAM | basic roles（`roles/owner`,`roles/editor`）のプロジェクト/データセット付与 |
| 2 | IAM | `allUsers`・`allAuthenticatedUsers` への公開付与 |
| 3 | IAM | テーブル/データセット単位でなくプロジェクト単位の広域付与 |
| 4 | 列 | 機密度カタログ（FR-1.1）で high/medium 指定の列に `policy_tags` 未設定（カタログ整合） |
| 5 | 列 | タクソノミー不整合（存在しないタグ参照 / 孤立タグ） |
| 6 | 監査ログ | Data Accessログの過剰有効化（対象外データセットまで有効） |
| 7 | 監査ログ | sink未設定 / 保持過大 / 除外フィルタ不在 |
| 8 | コスト | 大テーブルで partition/cluster 未設定 |
| 9 | コスト | `require_partition_filter=false` |
| 10 | コスト | expiration 未設定の長期テーブル |
| 11 | データセット | ロケーション逸脱 / default table expiration 未設定 / CMEK |

**A+観点＝上振れ（Bには含めない）。** 精度が不確実なもの。
- GA4 PII値混入検知: `page_location`/`page_referrer` のクエリ文字列や `event_params`/`user_properties` の値にメール/電話/生ID等が混入していないか（値の内容検査＝精度不確実のためBから除外）。
- 古いSAキー等の周辺観点。

### FR-5 レポート／是正生成（点検）
- リスク一覧を機械可読（JSON/CSV）＋自然言語レポート（Markdown）で出力。
- IAM過剰権限に対し推奨ロールとTerraform是正案を生成。
- **是正提供フロー＝「レポート＋是正ドラフト添付」（適用は顧客/自分が手で実施）を既定とする。** 受託の事故リスクを避け、GitOps未成熟な中堅企業でも成立させる。
- 自動適用はしない。**PR自動作成は将来オプション**（顧客がTerraformリポ/GitOpsを持つ場合に限り提供）。

### FR-6 インスペクタ最小権限ロール（点検）
点検モードは客先環境の IAMポリシー・`INFORMATION_SCHEMA`・Logging設定・データセット/テーブルメタデータを**読む**必要がある。ガバナンスツール自身が過剰権限を要求しては本末転倒なので、以下を要件とする。
- 点検実行用の**最小権限リードロール**を定義（詳細は [design-modules-wif-wiring.md](design-modules-wif-wiring.md) A-5）。書き込み権限は付与しない。
- ロール定義自体を再利用テンプレとして同梱（客先で付与しやすくする）。
- 是正の適用権限は含めない（適用は人手＝FR-5と一貫）。

### FR-7 案件パラメータの統一（再利用の要）
案件ごとに変わる値を単一の「案件パラメータ」に集約し、テンプレ本体を不変に保つ。
- 対象: 機密度カタログの上書き（FR-1.2）、ネスト展開キー（FR-1.3）、IAMのロール/主体マッピング（FR-2）、監査ログの高感度対象（FR-3）、コスト予算（構築レール側）、点検対象範囲（FR-4）。
- これにより「汎用アセット（保持）＋案件パラメータ（納品）」の切り分け（§2.5）を実装レベルで担保する。

### FR-8 検証環境（B実証用・カタログ網羅の最小セット）
本番データ量に依存せず、カタログの全機構を実演できる最小の疑似機密データを用意する。
- **high**: `user_id`（疑似の会員ID）、`event_params` に混入させたメール（→ 展開列としても用意）。
- **medium**: `user_pseudo_id`、`geo.city`。
- **展開デモ**: `event_params` の既知キーを型付き列に unnest し `policy_tags` を付与（FR-1.3）。
- **A+実演**: `page_location` にクエリ文字列PII（例 `?email=...`）を混入させた行を用意し、値スキャン検知を実演。
- これ1環境で「①列アクセス制御が効く／②unnest＋タグ付け／③PII混入検知(A+)」を通しで示す。

### FR-9 マートメタデータ説明（点検）

マート利用者が列の意味と取扱条件をBigQuery上で確認できるよう、点検モードは
BigQueryネイティブのテーブル／列 `description` の完全性を評価する。

| ID | 要件 | 優先度 | 判定 |
| --- | --- | --- | --- |
| FR-9.1 | `mart_patterns` に一致するデータセットと、安全側に倒して完全点検する未分類データセットについて、テーブル／ビューと全leaf列の `description` を点検する | Must | 欠落または空白のみならCHK-12 LOW |
| FR-9.2 | 生 `analytics_*` と明示除外データセットはCHK-12の対象外とする | Must | 行データと生テーブル列を読まない |
| FR-9.3 | 判定は既存の `bigquery.tables.get` 応答だけで完結し、BigQuery query jobを発行しない | Must | 追加API呼び出し0、BQ処理0 bytes |

CHK-12はFR-4の11セキュリティ観点に対する追加のデータガバナンス観点であり、
既存のAcceptance B分母（11観点中10以上）と過去証跡を変更しない。

descriptionの文章品質とデータ出所／lineageは対象外とする。出所管理が必要になった
場合は、descriptionのfree-textを解析せず、構造化された汎用lineage仕様を別Issueで
定義する。

## 6. 非機能要件
- **べき等性**: 構築IaC・点検は再実行で同じ結果（点検は副作用なしの読み取り専用）。
- **コスト**: 点検自体のBQスキャン量を抑える（`INFORMATION_SCHEMA`中心、フルスキャン回避）。
- **可搬性**: 複数プロジェクト／案件へテンプレ再利用可能（変数化）。
- **安全性**: 是正の自動適用はしない。生成物はドラフト、適用は承認ゲート経由。

## 7. アーキテクチャと既存資産統合

### 7.1 アーキテクチャ（想定）
- IaC: Terraform（taxonomy / policy tag / data policy / dataset / IAM / log sink）。
- 点検: スクリプト（`INFORMATION_SCHEMA` + IAM policy取得 + Logging設定取得）→ 観点評価 → レポート。
- 生成AI: 点検結果JSONを入力に、レポート/是正ドラフトを生成（決定論的な検出結果を"枠"として与える）。**実行基盤＝ローカル/CLI**（点検スクリプトの一部としてAPI呼び出し、運用インフラを持たない）。
- CI/CD: 構築IaCのplan/apply。**点検はオンデマンドCLI実行を既定**（定期実行=Cloud RunはA+の継続強制リコンサイラ拡張）。

### 7.2 既存資産との統合（4リポ実査に基づく確定版）

**配置＝「4段目の専用テンプレ」**。既存の3段チェーンに、本アセット専用テンプレを1段挟む。

```
ai-dev-foundation（万能）→ terraform-gcp-template（GCP層）→ 【secure-ga4-bq-template（本アセット）】→ 客先案件インスタンス
                                                        ↘ terraform-gcp-modules（BQモジュールをtag参照）
```

- 各客先案件は `secure-ga4-bq-template` から "Use this template" で作成。`.ai/` 規約・`.skills/`・canonical `make` 契約・`infra/envs/<env>/`・**module-by-tag参照** を（terraform-gcp-template経由で）継承する。
- **§2.5「汎用アセット保持＋案件パラメータ納品」の実体**＝4段目テンプレが汎用アセット、各インスタンスの差分が案件パラメータ。

**置き場所の原則（基盤を重くしないため）**: データスタック固有物は**上流（foundation / terraform-gcp-template）に押し上げない**。上流に足すと全プロジェクトがテンプレ作成時に一括継承して重くなる（foundation自身の COD-051 speculative-generality禁止にも反する）。
- **一括継承型ベース**（foundation/template の profiles・skills・規約）: 触らない。
- **オプトイン型ライブラリ**（terraform-gcp-modules / gcp-cicd-workflows）: 追加はtag/`uses:`参照で使う人だけが引く＝無害。ここは増やしてよい。
- **本アセット固有物**（dbt/Dataform profile・GA4カタログ・WIF配線）: **4段目テンプレの中だけ**に持つ。

**継承して再利用するもの**
- `ai-dev-foundation` の **セキュリティ規約**: `SEC-011`（GA4識別子=PII分類）・`SEC-021`（最小権限IAM）・`GR-030`（セキュリティ後退禁止）。本アセットのFR-1/FR-2/FR-6と直結。決定は ADR/`decision-log` に記録。
- `make` 契約（`build`=認証不要validate、`test-integration`=`terraform test`）と CI（`ci.yml`/`iac.yml`=Trivy/Checkov）。
- `terraform-gcp-modules//modules/github-oidc`（CIデプロイヤ認証＝WIF）。`service_account` を用途別に分けられる。

**新規に作るもの（どのリポにも存在しない＝本アセットの実装本体）**
- `terraform-gcp-modules` に **新規モジュールを追加**（既存規約: 1モジュール1ディレクトリ、provider宣言なし、validated inputs、`versions.tf` で google `>=5.0,<8.0`、tag固定 `?ref=vX.Y.Z`）: `bigquery-dataset` / `bigquery-policy-tags` / `bigquery-data-policy` / `log-router-sink` / `bq-inspector-role`。インターフェースは [design-modules-wif-wiring.md](design-modules-wif-wiring.md) 参照。
  - ※ 既存 `github-oidc` は**プロジェクト単位IAMのみ**なので、データセット/テーブル単位の最小権限は新規で作る。
- `gcp-cicd-workflows` に **新規ワークフローを追加**（house style: `workflow_call`＋WIF＋`setup-gcloud`）: `bq-cost-gate.yml`（dry-runバイト予算ゲート）／`bq-inspect.yml`（オンデマンド読み取り点検。`service_account` にインスペクタ最小権限SA〔FR-6〕を渡す）。
- SQL lint（SQLFluff/`dataform format`）は、gcp-cicd-workflowsの設計原則（cloud認証が要るものだけを置く）に従い **foundationの `make lint` 側**に置く。

**要配線（テンプレに未整備＝追加作業）**
- `terraform-gcp-template` は **WIF/apply自動化を配線していない**（CIは自己完結のmake呼び出し）。`github-oidc` モジュール＋`gcp-cicd-workflows/tf-apply` を明示的に配線する必要がある。
- テンプレは `gcp-cicd-workflows` を消費していない（別資産）。cloud認証が要る経路（apply・cost-gate・inspect）だけを繋ぎ込む。

**切り出し境界（客先納品）**
- 客先は社内モジュール/ライブラリを持たないため、module-by-tag参照を**内包版に差し替え可能**にする（`?ref=` git source → vendored に切替）。依存を明示し、納品パッケージを自己完結化できる状態を保つ。

## 8. 受け入れ基準（目標の達成基準への写像）
- **B（基準）**: 検証環境で統制3項目を組み込んだクリーン構築を再現し、疑似機密列で列レベルアクセス制御が効くことを実証。点検スクリプトが **FR-4のB観点を10項目以上** 検出できる（決定論・高信頼のみ）。成果物＝定義＋構築IaC＋点検スクリプト＋実証ログ。
- **A**: 実案件/本番相当へ適用（構築 or 点検カバレッジ100%＋是正提案）＋生成AIレポート/是正ドラフト自動生成。**A+上振れ**: FR-4のA+観点（GA4 PII値混入検知）／Cloud Run定期実行（継続強制リコンサイラ）。
- **S**: 2案件以上へ適用 or 部門標準採用、2案件目でテンプレ再利用の工数削減を実測、スペシャライゼーション実績提出。

FR-9／CHK-12はB達成後の追加ガバナンス機能として個別に受け入れ、上記レベル判定を
遡及変更しない。

## 9. 工数見積もり（アセット構築＝一度きりのR&D）

> 前提: 単独開発・AI支援あり。**「アセット構築工数（本節）」と「完成後の1案件あたり構築工数（目標の10〜15人日）」は別物**。本節は前者。人日はレンジで示す（精度の過信を避ける）。
>
> 重要な性質: **再利用は"周辺"を助けるが"中核"は縮まない**。CI・make契約・WIF・リポガバナンスは既存資産から流用できるが、価値の本体（BQモジュール群・点検エンジン・GA4カタログ）は既存資産にゼロなので、その工数は圧縮されない。

### 9.1 再利用による圧縮（周辺）
| 流用元 | 節約 |
| --- | --- |
| terraform-gcp-template（.ai規約 / make契約 / infra-envs / CI `ci.yml`・`iac.yml` / PR・issue雛形） | 約3〜5人日 |
| gcp-cicd-workflows（tf-plan/apply・PRコメント・db-migrate型） | 約1.5〜2.5人日 |

| モジュール規約の参照実装 | 約0.5〜1人日 |

→ 素で作れば45〜60人日規模のところ、周辺流用で**約7〜11人日を圧縮**。

### 9.4 実現可能性
7〜12月（①調査〜③検証＝B相当まで）は約6か月。本目標は重み40%なので可処分を仮に40%とすると約48人日が使える計算。**B（18〜28人日）は余裕を持って収まり**、Aの一部（AI生成・cost-gate）も前倒し可能。→ 目標のB較正は妥当（自力で確実に届く）。
- **3目標合算の整合**: 本目標B（18〜28）＋型化B（17〜28）＋AI環境B（17〜27）＝**52〜83人日**。期間（7〜3月・営業日約180日）の**約3〜5割**で、通常業務と並行して成立する範囲。※「重み%＝時間%」の簡易計算を3目標で単純合算すると稼働100%になってしまうため、実現可能性の判定はこの**合算絶対量**で確認する。

### 9.5 上振れリスク要因
1. 列レベルセキュリティの結合（タクソノミーlocation一致・タグIAM伝播・CLSが実際にブロックするかの検証）。
2. inspector custom roleの最小権限（§11未決）— 綺麗な既定read-onlyロールが無く実機で権限の過不足を反復。
3. 点検11観点の精度（誤検知/取りこぼしのチューニング）。
4. dbt/Dataform手本のパリティ（2エンジン目でハマるとS工数が膨らむ）。

## 10. リスク・対策
- 統制追加で工数が増えて見える → 価値を「同等工数＋2案件目以降の削減」に置く。
- 自社に実証データが乏しい → 検証環境に疑似機密列（本番データ量非依存、FR-8）。
- 監査ログのコスト増 → FR-3の4層設計。
- 実案件が期内に出ない → Bは検証環境＋自社点検で完結、実案件はA以上に限定。
- taxonomy/policy tag API の名称揺れ（Data Catalog→Dataplex→Knowledge Catalog）→ **統制自体（ポリシータグ/タクソノミー）は非推奨ではない**が、Terraformリソース名が追随して変わりうる。バージョン固定＋移行時の追随で対応。
